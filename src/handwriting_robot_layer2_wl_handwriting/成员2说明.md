# 成员2（wl）：个人笔迹处理

## 项目边界

输入为文字和用户笔迹档案，输出为统一 `StrokeDocument`。成员2负责字符采集、用户字库、缺字检查和笔迹轨迹生成，不直接控制写字机，不复制界面模块或设备通信代码，也不承担最终页面排版。

## 调整后的工作顺序

1. 建立第一阶段目标字符集：1000 个现代汉语常用汉字，并补充实验记录领域汉字、字母、数字和常用符号。当前清单共 1140 个不重复字符。
2. 提供个人笔迹采集程序：桌面端支持鼠标/数位板，手机端支持浏览器手指/触控笔；所有设备统一为归一化轨迹，支持草稿、断点续录和同字多变体。
3. 采集一至两名书写者的代表性样本：优先完成常用汉字，再按实际演示文本的缺字清单补录。无需为验证方案采集大量人员笔记。
4. 将已有字符样本导出为 `StrokeDocument`：使用随机种子选择真实变体，并加入受控的大小、倾斜和位置扰动。相同种子可复现，不同种子形成变化。
5. 研究少样本风格迁移：当前已实现风格特征档案和标准有序骨架的确定性几何变换基线，可用于处理骨架库覆盖的未录字符；复杂偏旁比例和特定笔画写法仍需后续生成模型，不能用简单随机扰动冒充模型效果。
6. 延后扫描图/OCR：作文和笔记扫描图需要单字分割、标注与人工校正，待数位轨迹流程稳定后再评估。

## 与原项目要求的对照

- 模块职责、统一轨迹接口和“不控制设备”的要求已满足。
- 仓库要求的 README、`__init__.py`、示例和测试已补齐。
- 原一期目标为每名书写者 500～1000 个样本字符；当前 1000 常用汉字清单可覆盖该目标，字母、符号和领域字作为补充采集项。
- “同字 2～3 种变化”可通过真实多变体加随机种子扰动实现，但随机扰动只适用于已有字符。
- “标准笔顺字形的风格变形”和“未录字符自动学习”仍属于模型研究任务，当前版本只会报告缺字，不会伪造学习结果。
- 扫描图处理从第一阶段延期，不影响先完成数位轨迹采集和 `StrokeDocument` 联调，但应在阶段汇报中明确说明。

## 当前交付

- `resources/target_charset_phase1.csv`：带类别和来源字段的结构化字符清单。
- `collector_app.py`：鼠标/数位板采集入口。
- `mobile_server.py` 和 `web/`：手机局域网采集服务及触控页面。
- `capture_protocol.py`：手机、鼠标和数位板输入的统一校验与归一化协议。
- `resources/style_probe_charset.json`：用于第一名用户数位板试采的 100 个代表汉字。
- `collection_report.py`：试采完成率、压力统计和异常样本初筛。
- `preprocessing.py`：生成动态分析、画布布局和标准字形三套清洗轨迹。
- `style_analysis.py`：从代表字提取用户布局、动态、压力、倾角和字形几何风格档案。
- `skeleton.py`：校验标准字形的笔顺、笔画边界、中心线坐标和数据来源。
- `style_generator.py`：把标准有序骨架按用户风格转换为 `StrokeDocument`，不控制设备。
- `hanzi_writer_adapter.py`：本地按需读取 Hanzi Writer Data 中心线，为 1008 个目标汉字提供骨架覆盖。
- `SampleStore`：草稿、正式样本、变体和用户清单存储。
- `analyze_coverage()`：覆盖率和缺字清单。
- `render_text()`：基于已有样本的可复现随机变化与 `StrokeDocument` 输出。
- `export_cli.py`：供其他成员从命令行调用。

具体安装、调用和数据格式见本目录 `README.md`。

1. 从最新 `main` 创建 `feat/wl-handwriting-日期-主题` 分支。

4. 推送后创建 Pull Request，附示例输入、示例输出、测试结果和缺字统计。
5. 每周汇报样本数量、覆盖率、已解决问题和下周计划。
